## Technical learnings (design facts to honour)

- **The serialization format is dual-purpose** - it is both the local autosave and (public-filtered) the multiplayer wire format, and it was designed that way on purpose. Keep new state forward-compatible; don't break the shape casually.
- **Hotseat and online are deliberately the same game, two transports.** Board model, serialization, public-board format, turn structure, victory points and oust rules are shared code. Diverge only where a transport forces it or a local-only shortcut is clearly better. Write a new table feature once against the shared model and it lights up in both.
- **Cards are positioned with a composite `transform`** (translate + rotate + scale on a shared parent). So you **cannot** animate `transform` directly without breaking placement - the entrance/flip animations use the separate CSS **`rotate`** property, which composes on top. **No `perspective`** is applied (the orthographic squash is the safe choice across the shared transform); true 3D would need a per-card inner wrapper that owns the perspective.
- **Victory points are clamped 0-10 in 0.5 steps**, the same way on client and server. The oust bounty is **+6 pool and +1 VP** to the predator.
- **Seating is turn order; predator = the previous live seat, prey = the next.** The bounty goes to the nearest non-ousted seat *backward* from the ousted player.
- **Oust bounty differs by transport, same outcome:** online is server-authoritative (+6 pool routed to the predator's client, +1 VP set server-side); hotseat resolves locally (it credits the predator's *stored* board's pool, and +1 VP in the roster). In hotseat the +6 lands on a stored board, so it is only seen once you switch to that player - worth saying out loud.
- **`VERSION` is informational on load** (no hard gate), so version bumps are save-safe - but keep client and server in lockstep to avoid the online mismatch warning.

- **When a VTES rule or game function is unclear, look it up - don't guess.** The official rulebook is at **vekn.net/rulebook** (the link Elysium ships in-app); card data and rules text come from KRCG (static.krcg.org).

- **L3 (helicopter) has TWO coordinate worlds, and most L3 bugs are a mismatch between them.** (a) **Opponent furniture** — every `.mat`, `#l3table`, `#l3timer` lives *inside* `#l3stage`, which carries `transform: translate(pan) scale(Z)` with `transform-origin:0 0`. They scale + pan as one rigid unit automatically; you set their *unscaled* slot coordinates and the stage does the rest. (b) **Your own live board** — `#z-ready`, `#poolWrap`, `#edge`, your cards live *outside* the stage on `#board`, and are placed manually by `setCenterFrameA`, which folds the same transform in by hand: `Px + Z*coord` for position **and** `scale(Z)` for size. The rule: an element either rides the stage (put it in `#l3stage`, touch nothing) or is hand-folded (then it needs BOTH the `Px+Z*coord` position fold AND a `scale(Z)` size fold — see the inverse-zoom pitfall).
- **`l2pub` already contains the L3 zoom.** `l2pub.s = baseS * Z` and `l2pub.ox = Px + Z*baseOx`. So anything placed through `pubXform`/`pubInv`/`l2pub.s` (ready cards, the pool globe, the Edge *token*) is automatically correct under pan+zoom — you do **not** multiply by `Z` again. Re-multiplying is the classic double-transform bug. Only hand-positioned elements that bypass `pubXform` (like the Edge *button*, positioned with raw `Px+Z*…`) must apply `Z` themselves.
- **`pubInv(pubXform(x)) === x` exactly** (verified): `pubXform` is `ox + s*(x-bx)`, `pubInv` is `bx + (x-ox)/s`. This is why the drag→drop→re-place round-trip for cards and tokens never snaps: drag draws at board-space `x`, drop stores `pubInv(x)` as canonical, re-place draws `pubXform(canonical)` = the same `x`. Trust this identity instead of adding correction fudge.
- **Geometric hit-testing beats `elementsFromPoint` across stacking contexts.** `opponentSurfaceAt(cx,cy)` / `readyZoneAt(cx,cy)` walk `.mat[data-seat]` (and `.matzone-r` for ready) comparing `getBoundingClientRect()` against the cursor. This is robust where `elementsFromPoint` fails — e.g. the Edge token lives on `#board` but the mats live in `#l3stage` (a different stacking context with `pointer-events:none` on the container), so `elementsFromPoint` returned the token or nothing. Card-give already used the geometric path; reuse it for any cross-board drop.
- **Mat-relative fractions are the coordinate-system-agnostic way to place something on another player's board.** When dropping onto an opponent's mat, never `pubInv` the drop point in *your* slot's space — that gives a wildly off canonical value (your slot and their mat are different screen regions). Instead take the drop point as a fraction `(clientX-matRect.left)/matRect.width` of the target mat, then map that fraction into the shared canonical box (`box.x + rx*box.w`). This is exactly how `giveTo` places cards (`rx/ry` 0–1) and how the Edge handoff lands the token inside the receiver's Ready.
- **Hotseat stores three layers per seat; a cross-seat change must touch all three.** (1) `net.hot.boards[seat]` is a **JSON string** (`serializeGame()`) for non-active seats — `JSON.parse` → mutate → `JSON.stringify`, never treat it as a live object. (2) `net.boards[seat].pub` is the rendered public mat view (what `buildMat` reads in the overview). (3) The **active** seat has neither — it is the live `state`, so skip its stored slot (it's stale while live) and mutate `state` directly. The Edge handoff/takeover writes edge-flag + token position into all applicable layers; missing one means the change shows in some views but not others.
- **A token only renders on its controller's mat if it's in `pub.tokens`.** Tokens (the Edge, and any future token) are live DOM on `#board` for the active player only. To make them visible on *every* board in the overview — and when you pass the turn in hotseat — `buildPub()` exports `tokens:[{type,x,y}]` and `buildMat()` draws them from `pub.tokens` with the same `X()/Y()/s` mapping as cards (read-only, `pointer-events:none`). Anything that should appear on opponents' boards has to flow through `pub`, not just the live token Map.
- **L2 Prey/Predator tabs: keep a single interaction model.** The tabs (`#l2preyTab`/`#l2predTab`, `position:fixed`, z:630) sit above the folded-out columns (`.l2pane2`, z:60) — the existing `toggleL2Fold` handler works in both directions. Hiding the tab when the column opens forces users to switch mental model from "click tab → open" to "click header strip → close". Showing the tab in both states keeps one consistent UX. The `l2tabon` class provides the open-state visual cue; the header strip (`l2hd`) remains as a parallel close path.
- **Token/globe drag in L4 must apply the same scale as placement.** `placeToken` and `placeL4Globe` multiply by `L4_TOKEN_S` (0.7), but the pointermove handlers originally used only `l2pub.s`. The missing multiplier caused a 1/0.7 ≈ 43% size jump at drag start — the token appeared to "lift" toward the cursor. Fix: `(l2pub.on?l2pub.s:1)*(l4on()?L4_TOKEN_S:1)` in both drag handlers, mirroring the card drag pattern (`L4_CARD_S`). General rule: any element scaled at placement must use the exact same scale expression during drag.
- **Hit-testing in L4 must use the visual scale, not the L2 base scale.** `attachTargetAt` computes card hit-boxes from `sc`; in L4, `sc` must include `L4_CARD_S` (0.5), otherwise the hit-box is 2× the visual card width and height. The same `(l2pub.on?l2pub.s:1)*(l4on()?L4_CARD_S:1)` pattern applies. Symptom: cards attach to targets that look far away, and "snap back" when the dropped card jumps to the false host's position.
- **L4 dock close AND scale should be deferred during hand-card drag.** In L3 the dock closes + card scales instantly at drag start (line of sight to the board matters). In L4 the board IS the dock area — closing and shrinking instantly removes context. Defer with `drag.l4DockDefer=true`: no `regrabAtScale` at start, card stays at s=1, dock stays open. Transition on cursor exit (`yd < H*0.66`): close dock, `regrabAtScale(c, drag, boardScale)`. Re-enter (`yd >= H-36`): `setL2Dock(true,true)`, `regrabAtScale(c, drag, 1)`. The transition checks must run BEFORE the x,y computation so that `regrabAtScale` offsets feed the same frame's coordinates — otherwise the card jumps one frame.
- **Stacked children should be invisible to placement logic.** Cards with `c.host` (attached to another card) should be filtered out of `attachTargetAt` and `nudgeFree`. Without the filter, children's hit-boxes cause false attachment triggers (the cursor overlaps a child, `topHost(t)` returns the host, and the dropped card attaches to the wrong minion), and `nudgeFree` pushes newly played cards away from the stack's visual footprint even though the children are "riding" their host. Children remain hoverable, previewable, and selectable — only their collision/overlap footprint is suppressed.
- **L3/L4 board AR is canonical (collapsed-panel proportions).** `l3canonAR` is computed once at enter-time based on the board width as if the aside panel were collapsed (+244px compensation if open). `l3geom()` letterboxes this AR into the current board dimensions: panel collapsed → fills edge-to-edge, panel open → dead space top/bottom. `_asideResize` flag distinguishes aside toggles (keep AR) from true window resizes (recompute AR). This keeps the `#l3table` marking proportionally fixed.
- **L3/L4 should use `l2cols` (fill `#tableArea`) to unify dead space.** The fixed 1004×616 canvas + CSS scale creates a letterboxed dead space with `#tableArea` background, separate from the board's felt-texture zoom dead space. Adding `l2cols` makes the board fill the container via CSS `inset:0; width:auto; height:auto; transform:none` — all dead space is inside the board. `l3geom()` returns the actual board dimensions; all L3/L4 layout is fraction-based and resolution-independent, so it adapts. `fitBoard` must return early for `l2cols` regardless of L3 mode (the old guard `l2cols && !l3mode` skipped L3). The resize handler must include `net.view==='l4'` → `renderL4()` — L4 was previously unhandled, leaving `l2pub` stale after a container resize.

- **Seats are array positions, not a stored field.** On the server a player object has **no `.seat`** - seats are computed only inside `roster()` as index+1. Address a seat by `room.players[seat-1]`, never `players.find(x => x.seat === n)` (that silently finds nothing). And the server **derives** the actor's seat from `room.players.indexOf(p)+1` rather than trusting a client-sent `bySeat`/`toSeat` - a client can lie about which seat it is.
- **You can drive the client's real functions in Node.** `elysium-test-harness.js` loads the whole `<script>` under a Proxy DOM stub (every property access returns a chainable stub; numeric-ish props return `0`) and exposes the functions. The monolith runs to completion because almost all DOM work lives inside function *bodies*, not at top level - so `serialize`/`restore`/`buildPub`/`pubXform`/`baseView`/`dbg` are all callable and assertable. `let l2pub` is a mutable object, so a test mutates its fields and the real `pubXform`/`pubInv` read them - no need to extract anything. This is what makes the refactor safe: the suite guards the *live* code, not a copy.
- **The client has a diagnostic channel now.** `dbg(tag, e)` feeds a capped ring buffer (the silent network/escrow catches + uncaught errors); **`elysiumDbg()`** dumps it from the browser console; `conv.debug:true` mirrors each new entry to `console.warn` live. When an online desync is reported, ask for `elysiumDbg()` output - the buffer records even with the flag off.

## Technical pitfalls (avoid these)

- **"Wired but never live-tested" rots silently.** The `edgePass`/`edgeTake` online verbs shipped with four latent bugs (wrong seat lookup, dropped `pos` field, trusted the client's seat, missing `!room.started` guard) purely because hotseat is the primary transport and nobody exercised the online handoff. Treat any online-only path that isn't run regularly as suspect until proven - it's a prime candidate for the logic suite plus a live two-client check.

- **Python `\b` i strängar = backspace, inte regex word boundary.** I en Python heredoc-sträng () tolkas  som ASCII 0x08 (backspace), inte den bokstavliga texten . Resultatet i JS-filen blir  som aldrig matchar. Fix: använd raw strings () eller dubbla backslash (). Denna bugg drabbade sect-deriveringen — alla fem regex-tester returnerade null, och  var alltid false, vilket gömde hela Follow Path-menyn.
- **The `\u` / `\U` backslash trap** - inside a `'PYEOF'` heredoc, a single backslash produces the character; a double backslash produces literal `\u...` text in the file. This caused a literal-escape bug twice (the trophy emoji). After any edit that adds special characters, grep the output to confirm the real character landed.
- **Line-number drift** - every edit shifts subsequent line numbers. Re-grep before the next edit; never trust a remembered line number.
- **Fragile find-anchors** - don't put smart quotes, em-dashes or arrows in find-strings; pick plain-ASCII substrings that are unique, and keep the fancy characters in the replacement only.
- **Don't run the WebSocket test suites blind** - the sandbox network is usually off, so a failed connection is not a code failure. The syntax check + the load harness are the dependable gates.
- **Don't claim runtime/visual behaviour works** - the sandbox can't render or run the game, so animations, layout, turn flow, hotseat switching and every online path are unverified. State that plainly and let Johan test live.
- **Don't forget to copy the freshly built file** - `cp` to outputs and `present_files`, and verify the *new* version went across (a stale-docx copy slipped through once; the timestamps/sizes caught it).
- **Don't relitigate dismissed ideas** - when an idea is set aside, the reasoning is recorded (a habit carried over from Abyssal Survivors). Check before re-proposing.
- **Inverse-zoom is the tell that an element has position-fold but not size-fold.** If something in L3 *shrinks as you zoom in* and drifts relative to everything else, it has `left/top = Px+Z*coord` (position scales) but a fixed pixel size (no scale). Fix: add `transform: scale(Z)` with `transform-origin:0 0` so size scales too. This bit the Edge **button** — position rode the zoom but it stayed 32px, so it looked like it was counter-zooming. (The Edge *token* never had this because it goes through `pubXform`+`cardTransform(…,s)` which scales both.)
- **`transition: all` + per-frame JS `left/top` = glide-and-snap-back.** Any element you reposition every animation frame during an L3 pan must NOT have a CSS transition on position. `#edge` had `transition:all .25s`, so each pan frame animated toward the new spot with a 250ms lag — it floated behind the cursor and snapped back on release. Fix: restrict the transition to visual props only (`background, border-color, box-shadow`), never `left/top/transform`. Same root as the `.l3panning .card{transition:none}` rule.
- **A read of a token's position must happen BEFORE `removeEdgeToken()`.** In the handoff, reading `tokens.get('edge')` *after* removing it returns null and the drop position is lost. Capture coordinates first. And note `tok.x/tok.y` is **stale during a drag** — it's only refreshed by `placeToken` on the *next* frame — so derive the live drop position from the drag's own `d.x/d.y` (board-space), not from `tok.x`.
- **A CSS rule with no inline-style counterpart wins silently.** `#board.l3mode #poolWrap .plabel{ display:none }` (left over from before the pool zone existed) hid the "Pool" label no matter what JS set, because the JS only touched `font-size/left/top`, never `display`. When an element "won't show" and the JS looks right, grep the stylesheet for a leftover rule on that selector before debugging the JS.
- **Calling a non-existent function is a silent bug until exercised.** `convertCardKind()` called `updateClan()` (which didn't exist) — passed syntax check and load harness because the call was never reached. The fix was `updateMyClan()`. Lesson: the load harness catches top-level crashes but not errors inside unreached function bodies. When renaming or moving functions, grep for all call sites.
- **One DOM element can't be in two places.** `#poolWrap` was being driven both as your slot's pool zone AND as the dock-panel pool, so the two writers fought (right size for one, wrong for the other). The fix was a dedicated second element (`#dockPool`) for the dock, with `updatePool()` syncing both. If you're toggling the same node between two contexts on every layout, you probably need two nodes.

## v1.5 learnings (the table-views interactivity pass)

- **`mpSend` is WebSocket-only - a no-op in hotseat.** `function mpSend(o){ try{ net.ws.send(...) }catch(e){} }`. In hotseat `net.ws` is null, so any action routed solely through `mpSend` silently does nothing. This is why the cross-table give failed offline. The fix is the recurring hotseat pattern: **apply locally** (`giveHot` injects into `net.hot.boards[seat]` + `net.boards[seat].pub`) instead of sending. Online keeps the server round-trip; the `gave`/`given` echoes are what remove/place the card there. Always ask "does this path work without a socket?" for any hotseat feature.
- **One give, two transports.** `giveTo()` dispatches: `if(net.hot && !inRoom()) giveHot(...) else mpSend({t:'give'})`. The hit-test (`opponentSurfaceAt`) and the call site (drag pointerup, the Reaction strip) are shared; only the commit differs. Same philosophy as the whole hotseat/online model - write once against the shared model, branch only at the transport.
- **Fan-rotated panel cards break drag-offset math.** Dock/hand cards are placed with a slight `rotate((i-mid)*2.2deg)` about their *centre*, but the grab offset is measured against the un-rotated top-left. When the card straightens (`rotate:0`) + scales + flips its transform-origin to `0 0` on drag-out, the offset is wrong and the card lands high. Clean fix: on drag-out, **centre the shrunk card under the cursor** (`drag.ox=CW*s/2; drag.oy=CH*s/2`) rather than scaling the stale offset.
- **A z-coverage bug surfaces as the quick-phrase menu.** When an overlay (the open L2 dock background, z:4400+) sits over an interactive element (the pool glob, no z), right-clicks hit the overlay; since it has no menu of its own, the global fallback fires and you get the quick-phrase menu instead of the element's. Fix = raise the element above the overlay (`#poolWrap` -> z:4700 when the dock is open), not touch the menu code. The badge-vs-zone z rule from before is the same family: a child's stacking is capped by its raised parent.
- **The home-view abstraction.** `baseView()` + `goHome()` unify "where a game opens and where Home/Esc return you", reading a local setting (`conv.defaultView`) unless a hosted game locks it (`net.tableViewLock`). Centralising it meant ~10 `switchView(null)` go-home sites became one `goHome()`.
- **Syncing a new host-locked setting = mirror an existing one.** To add a room-wide setting, copy the `reactSecs`/`matchEnd` path end to end: server stores `room.X` from `create`, forwards it in join/state + the `lobby` snapshot + `started`; client sends it in `create` and reads it on each of those messages. Additive + backward-compatible (old clients ignore the field). Server changes are unverifiable here - say so.

## v1.7 learnings (Played-tab polish, tab-centering)

- **`avRenderCard` returns the wrapper — use it for event listeners.** The function builds and appends an `.avc` wrapper to `host`, then returns it. Callers that previously ignored the return value can now attach `click` / `contextmenu` / etc. directly on the wrapper without a second DOM query. This is the right attachment point for per-card overlays like `#playedRow`.
- **Ownership check in a Played overlay: `state.cards.get(c.id)`.** Cards rendered by `renderPlayed()` come from two sources: live `state.cards` (mine) and `net.boards[seat].pub` (opponents'). The quickest ownership test is `!!state.cards.get(c.id)` — truthy only for cards currently in my active state. Opponent cards from pub boards are never in `state.cards`. (In hotseat the "active state" is the seat currently under control — same rule applies.)
- **Per-card "clear" = `avBaseline.add(id)`, not `snapshotPlayed()`.** `snapshotPlayed()` re-baselines *everything* currently in play (equivalent to "Clear all"). To remove one card from the Played display without touching anything else, just add its id to `avBaseline` directly, then call `renderPlayed()`. Simple and surgical.
- **CSS variables for `position:fixed` elements must be set on `:root` via JS, not inherited from `body`.** CSS variables normally inherit through the DOM tree, but `position:fixed` elements have the viewport as their containing block; in practice some browsers break the variable inheritance chain there (stacking contexts / intermediate composited layers are enough to cause it). Setting a class on `body` and relying on `body.aside-collapsed { --aside-w: 0px }` looked correct but didn't work live. Fix: call `document.documentElement.style.setProperty('--aside-w', value)` explicitly in `applyAside()` so the variable always lands on `:root` where fixed elements can reliably read it. The CSS class rule can stay as a fallback, but the JS call is what actually drives it. The math `--tab-mid: calc(var(--aside-w)/2 + 50%)` is correct (50% = 50vw for fixed elements); the variable just has to reach them.
- **Disambiguate bulk vs. per-item actions in the UI.** When a panel has a "Clear" button (bulk) and you add a right-click "Clear this card" (per-item), rename the bulk one to "Clear all" immediately — the two labels otherwise look equivalent to players who haven't read the docs. Same logic applies to any future Discard-all vs. Discard-one pairing.
- **`showMenu(items, x, y)` in a contextmenu handler: always `e.preventDefault()` first.** Without it the browser's native context menu fires on top of (or instead of) the custom one, especially on right-click inside an overlay. `e.stopPropagation()` is also good practice to avoid the board's own contextmenu fallback firing.

## v1.6 learnings (Played cards, the left-panel pass)

- **Scoped card CSS is a layout trap outside its scope.** `avRenderCard` sizes the `.avc` wrapper explicitly but relies on `#altview .avc{position:relative}` so the absolutely-positioned card *inside* (built by `addRCard`) lands in the wrapper. Render that card anywhere else (the new `#playedRow`) and the rule doesn't apply -> every card escapes to the overlay's top-left corner and stacks. Fix = mirror the rule for the new host (`#playedRow .avc{position:relative; flex:0 0 auto}`). When you reuse a render helper outside its original container, check which of its CSS rules were scoped to that container.
- **`showHoverFor` (the Ctrl-hover lupp) is name-based, not id-based.** It reads `el.dataset.cardName` + `imgUrl(name)`, not a card id or `state.cards`. So making a new card inspectable just needs `data-card-name` on a matched element, and it works for opponents' cards for free. The pointer tracker had to match the new wrapper (`closest('.card[data-card-name], .avc[data-card-name]')`). Stamp `data-card-name` only when the card is face-up, or you leak hidden cards.
- **Find the *universal* mutation point before hooking a counter.** The Played-cards badge was inconsistent because cards reach `ready` via more than one path (`playFromHand` *and* a board-level play) and the refresh hung on only one. `tagPlayed(c)` is the single point every play-to-ready hits - hooking the refresh there fixed it. A per-call-site hook is fragile; find the choke point (here `tagPlayed`, like `move`/`layout` elsewhere) and hook once.
- **ID-only keys survive hotseat take-control; composite keys don't.** The played-cards baseline first used `'me:'+id` / `'s'+seat+':'+id`. In hotseat, switching the active player flips a card between live `state` (`me:`) and a `net.boards` pub (`s+seat:`) while it keeps its id - so composite keys mismatch after the switch and every permanent flashes as "played". Keying on **card id alone** is stable across the switch (ids are seat-prefixed for demo decks, unique for given cards, preserved from saves). Online (`mySeat()` fixed) either works, but ID-only is the robust choice.
- **A custom pointer-drag needs `preventDefault()` or the browser steals it.** Without `e.preventDefault()` in the pointerdown, a mouse drag starts a native text/selection drag, which fires `pointercancel` and silently kills the custom drag - while taps/double-clicks still work (no selection). That signature (drag dead, tap fine) is the tell. Pair it with `user-select:none; touch-action:none` on the draggable.
- **The felt marquee must exclude every overlay that owns pointer drags.** The box-select pointerdown on the board fired on Quick-React / Alt cards (they live inside `#board`, `pointer-events:auto`) and trumped their drag. Any overlay with its own drag handlers (`.qrc, #qrbar, #altview`, plus `.card, .avc, ...`) has to be in the marquee's `closest(...)` exclusion list.
- **Sanitising relayed HTML while keeping one safe tag = placeholder-protect.** `sanRemote` strips all tags from opponents' log HTML, which also killed the `clog` card-links. Protect the one wanted pattern by swapping its opener for a sentinel (control chars U+0001 / U+0002), sanitising its id, stripping everything else, then restoring. Pure function -> unit-testable in the harness (link kept / `onerror`+`script` stripped / attribute-breakout neutralised).
- **The complete "refresh the current view" idiom has three arms:** `if(typeof net.view==='number') renderRemote(net.view); else if(net.view==='ov') renderL3(); else if(net.view==='l2') renderL2();`. `giveHot` only had the `ov`/`l2` arms, so a card given while visiting a single board (numeric `net.view`) didn't appear. Cover all three shapes after a state change (and call `renderAltIfOpen()` for the strips / Played tab).
- **CSS-hide over DOM-remove when a handler must survive.** Retiring a visible section (the "Card text" block) with `display:none` keeps its element - and its event handler - alive, avoiding a load-time throw from a now-missing `addEventListener` target. Removing the element instead means also removing its wiring. (To delete a block that *contains* unicode glyphs, anchor on surrounding ASCII and index-slice rather than matching the glyphs.)
- **A manual discard must re-check zone at move time.** `Discard mine` collects loose-ready cards, but a card may already have gone to ash another way; re-checking `c.zone==='ready'` inside the move loop makes it idempotent (no double-discard; also covers a duplicate in the list). Reading the *live* zone at click time, not a stale view, is what prevents the collision.

## v1.8–v1.10 learnings (auto-deal, discipline fix, anim speed, draw-click)

- **`dealOpening` uses setTimeout — not usable in a synchronous build loop.** The 80ms delay exists for animation effect in solo play. When building multiple hotseat boards in sequence, you need `dealOpeningSync()` (same body, no setTimeout). General pattern: if a function does the right thing but is async for cosmetic reasons, extract a sync sibling rather than fighting the async.

- **KRCG sends discipline names in two formats — trigram for old, full name for new.** Older disciplines arrive as trigrams (`"dom"`, `"aus"`); Oblivion and likely other post-V5 disciplines arrive as full names (`"oblivion"`). `DISC_NAMES` is trigram→fullname; `DISC_ABBR` (new, inverted) is fullname→trigram. Always index both forms in `cardTags()`. When a KRCG-dependent feature breaks for a new discipline, check which format its entry uses.

- **CSS `animation-duration` can be overridden per-element with inline style — no JS animation API needed.** Setting `el.style.animationDuration = '2.1s'` after a CSS class assigns its duration works cleanly. Apply it to *every* element that has its own animation declaration (`.cfx` wrap AND each `.cfximg` child), not just the top-level container.

- **JS-timed side-effects inside an animation must be scaled together with the duration.** `_swapAt` (the timeout that swaps the card image at the flip midpoint) is calculated as a fraction of the CSS animation duration. If you scale the CSS duration without scaling `_swapAt`, the image swaps at the wrong visual moment. Whenever a setTimeout is tied to an animation, treat it as part of the same parameter.

- **Browser fires click→click→dblclick. The two clicks must be suppressed manually in dblclick-mode.** There is no native "only fire on double-click, not on single click" event. The pattern: register both `click` and `dblclick`; in the click handler, guard with `if(conv.drawClick !== 'double')`. Ctrl+click (undo) is an exception — it should always work on click regardless of the mode.

- **Crypt draw is `influence()`, not `drawCard()`.** They look identical from the UI but `influence()` handles the pool cost prompt and the phase-counter (`state.phaseUsed.influence`). Never swap them. Library draw = `drawCard()`; crypt draw = `influence()`.

- **`c.owner` and `c.controller` are runtime card fields, not deck data.** `c.owner` drives oust routing (whose ash heap a card goes to). `c.controller` tracks current in-game control. Both are persisted in autosave snapshots but never touch the deck library. *Set owner* corrects a wrong owner; *Give control* changes controller. Completely separate concepts despite both being "who has this card".

## v1.11 learnings (click-cancel, actor FX, shuffle riffle)

- **`#cardFx` is `pointer-events:none` on the host, not on its children.** The CSS `pointer-events:none` is set on `#cardFx` (the container div), which prevents it from intercepting board clicks. But a `.cfx` element *appended inside* is a child in the DOM and is not covered by the parent's `pointer-events:none` — it receives pointer events normally. So a `click` listener on the `.cfx` wrap itself works fine for click-to-cancel without any extra CSS changes.

- **`{ once: true }` is the clean pattern for single-fire cleanup listeners.** When adding a cancel-click that races with `animationend`, `{ once: true }` ensures whichever fires first also removes the listener. No manual `removeEventListener` needed.

It returns `topHost(c)` — which is the host card if `c` is attached, or `c` itself. This is the correct object to pass as `opts.actor` to `cardFx`.

- **`playOnActive` had actor and target roles swapped in the play-FX call.** It called `cardFx(c, 'play', { target: a })` where `a` is the anchor minion — showing the minion's card image as a *target* (right side, smaller). The anchor is the *actor* (left side, caption). The fix is `{ actor: a, target: curTargetCard(c)||undefined }`. The log already said it correctly ("Govern played on Nergal") — the animation just wasn't matching.

- **`row.insertBefore(newEl, referenceEl)` is the right tool for prepending into an existing flex row.** The main card image (`img`) is already appended to `row`. To insert the actor image before it, `row.insertBefore(ai, aar)` and `row.insertBefore(aar, img)` in sequence gives `[actor-img] [arrow] [main-img]` without rebuilding the row.

- **Pile element IDs for future reference:** Board piles: `#z-crypt`, `#z-library`, `#z-ash`. Dock piles (L2/L3): `#dockCrypt`, `#dockLib`, `#dockAsh` (class `.qrlib` inside `#dockBody`). The dock is hidden/shown via `l2dockopen` class on `#board`. `overflow:visible` on `.sfx-wrap` lets card silhouettes stick up above the pile element boundary without clipping.

- **CSS custom properties per nth-child are the right tool for staggered parametric animations.** Instead of three separate keyframe rules or inline JS per child, set `--sfx-tx` and `--sfx-r` via `.sfx-card:nth-child(N)` rules and reference them inside a single shared `@keyframes`. Clean, no JS per card.

- **Guard against stacking animations on the same element.** If `shuffleFx` is called twice quickly (e.g. bulk shuffle-in of many cards), a second `.sfx-wrap` would layer on top. Check `el.querySelector('.sfx-wrap')` and return early if already running.

- **`animationend` fires per-element, not once for the whole subtree.** When the wrap contains multiple animated children, `animationend` fires for each child separately. To detect "all done", listen on the *last* child (`.sfx-sheen` in the shuffle FX, since it is appended last) or use `e.target===specificEl`. - **Online shuffle: fire FX locally, don't wait for a server echo.** `shuffleZone` sends `mpSend({t:'shufPile',...})` and returns early. The server shuffles its side; the client has no card-order knowledge to re-render anyway. Fire `shuffleFx(z)` immediately on the local side for instant feedback, independent of the round-trip.

## v1.12 learnings (SFX library wired to events)

- **`setLocked` is the one hook for lock/unlock SFX — not the call sites.** Multiple paths call `setLocked` (context menu, double-click, `activatePhase` via `unlockAll`, autoLock). Only `setLocked` fires the per-card SFX; `unlockAll` bypasses it intentionally (group unlock stays silent). Hook at the choke point, not the call sites.

- **`log()` SFX needs a cooldown, not a per-category gate.** Many game events call `log()` in rapid succession (oust fires 3–4 log lines, a pass fires 2). A simple `_sfxCool` boolean with `setTimeout(320ms)` prevents stacking without needing to know which specific event is firing. The "dominant" sounds (oust, pass) fire before the log line, so they win the auditory attention; the Warded sound just adds texture to quieter events.

- **`bell()` was called by `pass()` but defined as a standalone primitive — safe to replace.** The existing `bell()` + `strike()` were small self-contained helpers. Since the SFX library functions are added after `sfxPlay`, and `bell()` appears earlier in the file, the primitives were hoisted into the `sfxMakeReverb` etc. functions placed with the SFX library — called at runtime, not definition time, so order doesn't matter.

- **The `heartbeat()` replacement keeps the same function signature.** `startYourTurn` calls `heartbeat()` in an interval. Replacing the body of `heartbeat()` rather than the call site means no other changes needed. The old `thump()` helper (used only by heartbeat) was removed in the v1.99.28 dead-code sweep.

- **`showSay` "Hold on…" is SAY index 4.** `const SAY=['No block','Block!','No reaction','It resolves','Hold on…','Yes','No','Pass']` — index 4 is "Hold on…". The hook is `if((i|0)===4)` inside `showSay`, checked after `if(!phrase) return`.

- **`sfxPlayCard()` goes in `tagPlayed`, not in `playFromHand` or drag handlers.** `tagPlayed` is the universal play-to-ready choke point — every play path hits it. Same reasoning as the badge refresh.

## v1.17 learnings (hand zone contextmenu — djupare fix)

- **Alla kort-element är barn till `#board`, inte till `#z-hand`.** Korten appendas via `board.appendChild(el)` i `makeCard()`. ## v1.19 learnings (conv TDZ-krasch)

- **`let`/`const` i temporal dead zone kraschar även med `typeof`-skydd.** `typeof conv` kastar fortfarande `ReferenceError` när `conv` är deklarerad med `let`/`const` men ännu inte nådd under körningen. Bara `var` (som hoistas till `undefined`) är säker att läsa innan deklarationsraden. Konsekvens: om en variabel deklareras med `let`/`const` och refereras i en funktion som anropas top-level FÖRE deklarationsraden → krasch.

- **`applyHelper()` anropas top-level tidigt (fas-bar-bygg).** Kommentaren på `net`-deklarationen (`// declared early: the load-time phase-bar build calls applyHelper()→hx()`) finns just för att dokumentera detta. - **Rätt fix: flytta deklarationen, inte skydda call-sites.** Att lägga `typeof`-skydd i `hx()` funkar inte (TDZ). Att flytta hela `conv`-blocket till direkt efter `net` är den rena lösningen och matchar mönstret som redan finns (`net` är tidig av samma anledning).

Inga event-lyssnare hade registrerats för dem heller.

- **`localStorage.clear()` hjälper inte vid JavaScript-krasch.** Felet var i koden, inte i sparad data. Om klienten kraschar vid laddning är localStorage alltid oskyldig tills annat bevisas.

## v1.19–v1.20 learnings (reveal, menus, log hover)

- **The context menu system was only 2 levels deep — one shared `#submenu` panel.** A sub-item in the submenu rewrote the SAME panel, destroying the level-2 items and dropping the pointer outside all panels (→ pointerdown closes everything). Three-level menus (Reveal…, Return to library…) need a real third panel `#submenu2`. The clean generalisation: `_childOf(el)` maps each panel to the one below it; the original "hover a leaf closes the child panel" rule extends naturally to level 3.

- **Don't fix menu hover bugs with timers or pointer-events juggling.** Two iterations of close-delay timers + `pointer-events:none` toggling on lower panels caused new bugs (menu stuck on level 2, level 3 still suppressed). The actual problems were just (a) no third panel and (b) z-index. Adding `#submenu2` + raising z-index solved everything with no behavioural hacks. Match the original's simple model rather than inventing complexity.

- **`.card.handhover` is `z-index:4800 !important` and `#poolWrap` in the L2 dock is 4700.** Any floating UI that reaches over the hand cards or pool must sit above 4800. The context menus were at 700/710 and got buried whenever a submenu extended over the hand. Menus now live at 8000/8010/8020 (below `#logInspect` 8600, `#hoverCard` 9000, `#cardTip` 9999). When a panel "disappears behind" cards, check the card/pool `!important` z-indexes first.

- **`findInspect` keyed on `data-cid` (card ID) only — name-only log links didn't preview.** Revealed cards aren't in the recipient's `state.cards`, so their log links use `data-name`. The fix: a synthetic-card fallback (`synthetic:true`) built from the name, rendered as a plain image (no tokens/stack, since it's not a live card). Any future log link that references a card the viewer doesn't own should use `data-name` and rely on this fallback.

- **All three log-hover paths must match the same selector.** `#log` (ctrl/panel hover + click + outside-click guard) and `#logPeek` (Shift-log) each had their own `.clog[data-cid]` listener. Adding `data-name` support meant updating ALL of them to `.clog[data-cid],.clog[data-name]` — missing one leaves a dead hover zone.

- **`log()` now takes a third `privateLine` arg.** `localOnly` (arg 2) stops network relay; `privateLine` (arg 3) additionally keeps the line out of `curLines` → never archived to `turnLogs`, never in the `.txt` download, never autosaved. Use both (`true, true`) for private per-player info like reveals. The shared/saved log should only ever contain public lines.

- **`cardFx` kind='handreveal' must NOT broadcast via `t:'fx'`.** The broadcast guard is `kind!=='handreveal'`. Reveal routing is private (only recipients should see the cards), so it goes through the dedicated `t:'revealHand'` verb instead. If you add a new private animation, guard it out of the generic fx broadcast the same way.

- **Hotseat has ONE shared table log — there is no per-seat log.** `setActivePlayer` restores boards with `keepLog:true`, and `restoreGame` with `keepLog:true` deliberately doesn't touch `curLines` or the visible `#log` (the "shared table log" comment marks this). Consequence: you cannot write a line into "another seat's log" in hotseat — switching to them won't show it. An attempt to inject into `net.hot.boards[seat]`'s stored `curLines` is a no-op because keepLog never reads it back. This is why per-recipient reveal lines can't be reproduced in hotseat.

- **Hotseat is the online testbed, but reveal is a known divergence point.** Johan develops against hotseat to avoid spinning up multiple online clients. Most table features mirror faithfully (shared model). But reveal's per-recipient private lines are a place where hotseat shows the SENDER's view, not each recipient's. When a feature's correctness depends on what a *specific recipient* sees, that's the case to verify online with real eyes — hotseat can't show it.

- **A blocking `prompt()` loses the cursor position.** Reveal's "Random X…" asks for a count via `_promptN` (a `prompt()`), then needs to open the destination menu — but the mouse position is gone by then. Solution: open the follow-up `showMenu` centred (`innerWidth/2, innerHeight/2`) rather than at a stale cursor coordinate.

## v1.21 learnings (pool glob z-index vs the collapsed dock)

- **A scoped z-index lift can leave the *other* state broken.** `#poolWrap` was raised to 4700 only under `.l2dockopen`. That fixed the open-dock overlay (dockbg 4400) but left the collapsed state on auto z-index, *under* the dock handle `#l2dock` (4410). When a floating control "works in one dock state but not the other", check whether its z-index rule is gated on `.l2dockopen` and whether the *other* state has a high-z element (the handle) sitting over it.

- **The collapsed dock handle `#l2dock` (z-4410) is a click sink, and it's NOT in the board bail-list.** `board`'s contextmenu listener bails on `.card,...,#poolWrap,#z-*,#edge,...` but not on `#l2dock`. So any right-click that lands on the handle (including where it overlaps the pool zone) falls through to `openSayMenu`. If a zone's own right-click menu is being replaced by the quick-phrase menu, suspect an un-bailed element stacked on top — not a missing listener.

- **`openSayMenu` wins by elimination, not by intent.** The pool's own contextmenu listener calls `preventDefault()` (no `stopPropagation`), and the `document` fallback bails on `e.defaultPrevented`. So the say menu can only appear over the pool if the event never reached `#poolWrap` at all. "Wrong menu showing" ⇒ "the event is hitting something else first," not "two menus are racing."

- **There is exactly one pool glob.** `#poolWrap` is the only pool element; the dock has no pool tile. A perceived "second drawn glob" when the dock opens is `#l2dockbg` (z-4400, bottom 32%) showing through the framed pool box. Don't go hunting for a duplicate DOM node.

- **Pick the z-index that fixes the bug without changing unrelated stacking.** For L3 the minimal-risk value was 4500 (above handle 4410 + dockbg 4400, below dock cards 4600) — fixes collapsed-dock interactivity while keeping the pool *below* the dock cards when the dock is open, so the change can't newly obscure a card. Reach past the obstacle, not past everything.

## v1.23 learnings (matching the opponent pool globe to your own)

## v1.24 learnings (opponent pool menu + the pool authority model)

- **`.matpool` is a pure behavioural hook (no CSS).** Adding it + `data-seat` to any element makes that element open the opponent-pool capture-phase menu. Safe to stamp on the globe, the pool zone box, and the L3 pool value — no visual side effects beyond a cursor/title you set explicitly.

- **THE POOL AUTHORITY MODEL (important for any future host/pool work): pool is 100% client-authored.** The server stores VP (roster) and `out`, but never pool. The oust bounty's +6 is applied by the *predator's own client* when it receives `{t:'bounty'}` — the server only routes the signal. Consequence: anything that needs to change *another* player's pool online must be relayed to *that* client to apply locally and rebroadcast; it cannot be set server-side. ## v1.29 learnings (play with open hand — standing grant)

- **A "standing reveal" is a subscription, not an event.** The one-shot `revealHand` fires once; the open-hand grant must re-push on every hand change. The clean hook is `schedulePush()` — the single universal "my board changed" point that already debounces the board push and `pushEscrow()`. Adding `refreshOpenHands()` there means every existing hand-mutating path (draw, play, discard, give, undo) re-pushes for free, with zero new call sites.
- **`'all'` as a sentinel beside seat numbers keeps the grant list one-dimensional.** `state.openHand` mixes seat ints and the string `'all'`; `'all'` supersedes (granting all replaces the array with `['all']`). `openHandIsAll()` / `openHandHas(seat)` hide the check. Persists trivially in JSON (filter on restore: `v==='all'||Number.isFinite(v)`).
- **Grants belong in the save blob but NOT in the undo snapshot.** `serializeGame` includes `openHand` (survives save/load + hotseat swaps); `snapshot()` (undo) deliberately omits it, because undoing a card move should never silently revoke or re-grant a social permission. Two serializers, two scopes — don't reflexively add new state to both.
- **Hotseat reads the grant straight from the stored blob; online needs routing.** In hotseat every non-active board lives as a serialized blob in `net.hot.boards[seat]`, so the grantee parses the granter's `openHand` + `zones.hand`→`cards` locally — no transport. Online, the hand never sits in the public board (`buildPub` omits the hand by design), so the contents must be routed privately, exactly like `revealHand`. One feature, two resolution paths in `openHandCardsFor(seat)` — branch on `localTable`.
- **Reusing the shared menu builder means new entry points come free.** `oppPoolMenuItems(seat)` is already the single source for the pool globe/zone menu AND the Table-panel row (v1.26). `unshift`ing the 👁 Inspect item there lit it up in both places at once — the dividend of the v1.26 single-sourcing.
- **The open-hand grant is a scoped hole in the escrow secrecy — name it as such.** Normally the hand leaves the client only as an encrypted blob the server can't read. An active grant sends clean card names through the server (like `revealHand`). That's the intended behaviour, but it means the escrow guarantee is suspended *for the granted hand while the grant is live*. Worth stating to Johan rather than letting it look like the escrow model was weakened.

## v1.36 learnings (fly-to-pile animation)

- **Capture the source rect BEFORE `move()` — the card's DOM element is hidden/repositioned by `layout()` the instant it changes zone.** `flyToPile(c)` must read `c.el.getBoundingClientRect()` while the card is still in its old position. So every call site calls `flyToPile(c)` *before* `move(c, pile)`, never after. The clone is an independent `document.body` child that outlives the original element's relayout.
- **A "fly from A to B" effect is a different mechanism from `cardFx`.** `cardFx` is a centre-screen presentation overlay (the card appears big in the middle); it has no source/destination geometry. A travel animation needs its own clone positioned at the real on-screen source and transitioned to the real destination — they don't share code and shouldn't.
- **`offsetParent!==null` is the cheap visibility test for "which representation is on screen".** The library has four DOM representations across views (`z-library`, `dockLib`, `avLib`, `qrLib`); only one (or none) is visible at a time. `offsetParent===null` catches `display:none` and detached elements; pairing it with a non-zero `getBoundingClientRect()` is enough to pick the live one without tracking view state.
- **Double-`requestAnimationFrame` is the reliable "start frame paints, then transition" idiom.** Setting the start styles and the end styles in the same frame makes the browser skip straight to the end (no transition). One rAF isn't always enough under load. Two nested rAFs guarantee the initial position/opacity is committed to a paint before the transition target is applied.
- **`opts.noFly` keeps drag-returns clean while sharing the function.** Rather than branch the animation decision at every call site, `returnToPile` takes an opt-out. Drag paths (`dropCard`/`groupDrop`) don't call `returnToPile` at all (they call `move` directly), so they're already clone-free — but the flag documents the intent and is there if a drag path ever routes through `returnToPile`.
- **`transitionend` + `setTimeout` safety net is the standard cleanup pair for CSS transitions.** If the element is interrupted or the transition never fires (e.g. zero-distance, or `prefers-reduced-motion` collapsing it), `transitionend` may not arrive. A `done` guard prevents double-removal when both fire.

## v1.35 learnings (library pile drag)

- **`preventDefault()` on `pointerdown` suppresses downstream `click`/`dblclick` — compensate explicitly.** The drag handler calls `e.preventDefault()` on pointerdown to prevent text selection and scroll-jitter. This silences the click/dblclick events that the existing library listeners depend on. For the hand-drop case `drawCard()` is called directly in `onUp`. For a sub-threshold tap (no movement → `!ld.started`) the handler currently treats the release as cancel, leaving the library silent for that gesture. Acceptable on mouse; potentially feels dead on touch. Fixable by detecting `!ld.started` in `onUp` and calling `drawCard()` there too — deferred.
- **`pendingRevealDrop` pattern: set before the network send, consume on first read.** Online park-at-position needs to attach coordinates to a round-trip network message (`reveal` → `revealed`). Setting a module-level variable before the send and nulling it on first read in the handler is the minimal pattern — no new protocol field, no server change. Only safe because `reveal` responses are one-at-a-time per player (the server processes serially). Would not be safe if two concurrent `reveal` calls could race.
- **Ghost-card positioning: `transform:translate(-50%,-60%)` offsets the image up and to the left of the pointer.** This keeps the pointer tip visible below the ghost card rather than hidden under it. The -60% vertical offset (slightly more than half the card height) places the card's lower edge near the pointer, so the player can see what's underneath. Adjust the vertical fraction if the feel is off.
- **The 5 px threshold separates drag from click at the pointerdown level — not the click level.** The threshold is checked in `onMove` before revealing the ghost. This means a true drag is unambiguous from the very first pixel of movement past 5 px, and a stationary tap never shows the ghost.
- **`setPointerCapture` ensures the ghost tracks the pointer even when leaving the element.** Without capture, `pointermove` events stop when the pointer leaves `#z-library`. With capture on the element + global `pointermove` listener on `document`, movement is tracked until `pointerup` anywhere.

## v1.38–v1.43 learnings (crypt feature parity + drag guards + attachment zones)

Att mata tillbaka `c.tx/c.ty` till `place()` orsakar dubbel-transform. Spara alltid pre-transform-koordinater separat (`c.lx/c.ly`) när de behövs för senare `place()`-anrop (t.ex. `restack`). Regeln: `c.x/c.y` för ready (kanoniska), `c.lx/c.ly` för torpor/uncontrolled (pre-transform layout-koordinater).

**`z-index: N !important` i CSS slår alltid igenom inline `style.zIndex`.** `.card.dragging { z-index: 5000 !important }` kan inte överridas med `el.style.zIndex=4950`. Enda sättet att styra visuell ordning är att kontrollera *vilka element* som får `.dragging`-klassen — inte att sätta ett högre inline z-index. Lösning: flag (`drag.hostGroup`) som suppressar `.dragging` på barn, hosten får klassen exklusivt.

**KRCG:s korttyper är enkla strängar, aldrig slash-komposita.** Den officiella listan: `Action`, `Action Modifier`, `Ally`, `Combat`, `Conviction`, `Equipment`, `Event`, `Master`, `Political Action`, `Power`, `Reaction`, `Retainer`. Multi-typ-kort har *flera element* i `card.types[]`-arrayen. Ord-splittning på sammansatta strängar är fel approach — direktuppslag på hela strängen är rätt.

**Sortering som muterar zonen in-place täcker alla renderingsvägar.** `renderHand()` sorterade en lokal kopia vilket bara fixade `#handRow`. `layoutZone('hand')` använde `state.zones.hand` direkt. Regel: om en zon ska ha persistent ordning, sortera `state.zones[z]` in-place vid varje move till zonen (i `move()`), inte i rendering-funktionen.

**`drag.group` kan sättas i `pointermove` även för single-card drag.** Ursprunglig design antog att `drag.group` bara byggs vid `pointerdown` (box-select). Det går lika bra att bygga det i `pointermove` (första rörelsen) baserat på `c.attached` — `drag.group`-loopen i rörelsefasen hanterar det identiskt oavsett ursprung. `drag.hostGroup`-flaggan skiljer de två fallen åt vid behov.

**`keepChildren`-flaggan i `move()` är nödvändig för host-drag till ready.** COUNTER_ZONES-skyddet (`!COUNTER_ZONES.includes(zone)`) bevarade barn vid torpor/uncontrolled men inte vid ready→ready. Explicit `keepChildren:true` + manuell uppdatering av barnens `x/y` i canonical-space ger korrekt beteende i alla zoner.

**Barnens kanoniska koordinater måste uppdateras relativt hosten vid ready-drop.** `releaseChildren` sätter barnens `x=tx, y=ty` (gammal position). Vid `keepChildren` uppdateras inte barnens `x/y` automatiskt — gör det manuellt med `readyToCanon(X + CX*(k+1), Y - CY*(k+1))` per barn. Utan detta hamnar barnen på fel kanonisk position och hoppar till sin gamla plats vid nästa layout.

## v1.55 learnings (hand panel sort)


- **Sort the display array, never mutate the zone array.** `state.zones.hand` is the canonical order (draw order, serialization). `renderHand` builds a separate `cards` array from it — sorting that copy gives correct visual order without corrupting the underlying model.

## v1.79 learnings (vektorisering av symbol)

**PNG → inline-SVG: raka linjer slår Bézier för skarpa spetsar.** En triskelion med spetsiga uddar vektoriserades bäst med `approxPolyDP` (polygon-approximation) + `fill-rule="evenodd"` snarare än kurv-anpassning — Bézier rundade av de skarpa spetsarna. cv2 `findContours(RETR_CCOMP)` ger ytter- och innerkonturer (hål), och evenodd renderar hålen korrekt. Resultat: 97.4% IoU mot originalet, ~3KB path. Generellt: för logotyp-/symbolgrafik med skarpa hörn, föredra polygon-approximation framför kurvor.

**Återanvändbar SVG-helper som returnerar en sträng.** `vtesSymbol(opts)` returnerar en `<svg>`-sträng (inte ett element) med parametrar för size/fill/opacity/cls/title. Strängformen gör den lätt att bädda in både i `innerHTML`-bygge och i andra SVG:er (t.ex. token-trianglen i v1.82). `fill:'currentColor'` som default låter CSS styra färgen där det passar.

## v1.80–v1.82 learnings (Edge-token: vad som INTE funkar + Token-familjen)

**Ett icke-kort-objekt får INTE läggas i `state.cards`/`state.zones`.** v1.81 gjorde Edge till ett `kind:'edge'`-kort för att ärva drag/placering/skalning gratis. Det **bröt hela korthanteringen**: klonanimationer slutade köra, dragna kort hoppade långt under muspekaren, och kort kunde släppas var som helst (zon-snäpp borta). Den exakta mekanismen fastställdes inte live, men lärdomen är hård: kort-pipelinen (drag/drop/animationer/serialisering/buildPub) antar att allt i `state.cards`/`state.zones` är ett riktigt spelkort. Objekt som inte är kort ska ligga i ett **eget system** vid sidan om.

**Men `place()`- och `bindCard`-matematiken kan ÅTERANVÄNDAS för fristående objekt.** Lösningen (v1.82) var en separat `tokens`-Map + egna funktioner som *kopierar* den rena matematiken utan att gå via kort-systemet: `placeToken` använder `pubXform`/`cardTransform` (samma skalning som ready-kort), och `bindTokenDrag` använder samma `f`-skalfaktor och `ox/oy`-offset som `bindCard`s pointermove — men HOPPAR ÖVER zon-/`move()`/`dropCard`-logiken som kräver `state.zones`-medlemskap. Drop lagrar bara kanoniska koordinater via `pubInv`. Resultat: identiskt drag/placerings-beteende i alla vyer, med kort-systemet helt orört.

**`cardTransform(x,y,0,s)` reduceras till translate+scale från origin.** Med `rot=0` tar `translate(cx,cy)…translate(-cx,-cy)`-paret ut varandra (rotate(0) är identitet), så uttrycket blir `translate(x,y) scale(s)` med `transform-origin:0 0`. Det betyder att `CW`/`CH` (kort-dimensionerna) är irrelevanta — funktionen fungerar för ett element av godtycklig storlek (token är 100px, inte kortstorlek). Bra att veta för all återanvändning av kort-transform på andra element.

**Token-drag-hopp: cachad display-position måste fräschas efter drag.** Buggen: muspekaren "flög iväg" när man drog Edge flera gånger i rad, men växling kort↔Edge fixade det. Orsak: `tok.tx/ty` (post-transform display-koordinater, som pointerdown beräknar greppet mot) uppdateras bara av `layout()`/`placeToken`, INTE av själva dragget. Efter en token-drag satte pointerup bara `tok.x/y` (kanoniskt) men lämnade `tok.tx/ty` inaktuella → nästa grepp räknades mot fel position. Kort fungerade för att `dropCard` → `layout` körs efter varje kort-drop och fräschar `c.tx/ty`. Fix: anropa `placeToken(tok)` i pointerup (efter att `tok.x/y` satts), analogt med hur kort layoutas om efter drop. Generell regel: om ett objekt cachar en display-position som greppberäkningen läser, måste den cachen uppdateras efter varje drag, inte bara vid global omritning.

**Ingen CSS-transition på draggbara element → omedelbart drag.** Edge kändes "segt" i v1.81 för att den ärvde kortens `transition` på transform (avsedd för animationer). `.token` har ingen transition + `will-change:transform` → dragget följer muspekaren omedelbart. Lärdom: draggbara objekt ska inte ha transform-transition (den gör dragget laggigt); animera istället bara vid diskreta tillståndsbyten om det behövs.

**Patch-revert är säkert tack vare write-after-loop.** Patch-skripten gör alla `s.replace` in-memory i en loop och skriver filen FÖRST efter loopen. Ett `assert s.count(old)==1` som failar mitt i loopen avbryter alltså innan `open(...,'w')` — filen på disk lämnas helt oförändrad. Det gör en misslyckad patch riskfri (ingen halv-applicerad fil) och en medveten revert enkel att verifiera.

**Token-familj-mönster (utbyggbart).** En `TOKEN_DEFS`-registry där varje typ deklarerar `label`/`size`/`svg(sz)`/`menu(tok)` + en generisk motor (`makeToken`/`placeToken`/`bindTokenDrag`/`tokenMenu`/`layoutTokens`) gör att en ny token läggs till med en enda post — den får drag, placering och egen högerklicksmeny automatiskt. Edge är första medlemmen; framtida tokens (blod, röst, etc.) kräver ingen ny infrastruktur.
## v1.83 learnings (Give control submenu)

**`prompt()` ska bytas ut mot submenyer där listan är känd.** `Give control…` var det sista `prompt()`-anropet i kortmenyn. Spelarlistan är alltid känd vid meny-öppning (`net.roster`), så den passar naturligt som undermeny. Mönster: bygg `_sub = []`, filtrera roster, push en post per spelare. Om listan är tom: använd leaf-item med `toast()` istället för ett tomt `sub:[]` (en tom submeny ser trasig ut).

## v1.85 learnings (settings export/import)

## v1.87 learnings (log-target + namnbaserat target)

**Tre semantiskt distinkta target-fall kräver tre grenar.** Eget kort (data-cid, mine→setTarget), motståndares kort på brädet (data-cid, remote→setRTarget), namnbaserat (data-name→setNameTarget). Att försöka unified dem i en funktion leder till fel sekretess-logik. Separata funktioner är rätt mönster.

**`net.rtarget.cid===null` som diskriminator för "namnbaserat target".** Befintlig outline-rendering kollar `net.rtarget.cid` → ingen cid = ingen outline. Det betyder att ett namnbaserat target smidigt återanvänder `net.rtarget`-strukturen utan att kräva ett nytt fält, och target-knappens tooltip fungerar direkt (läser `r.name`).

**`data-from-seat` som bakåtkoppling i privata loggrader.** Mottagarraden för reveal har `data-name` men inget `data-cid` — utan `data-from-seat` kan högerklicksmenyn inte koppla kortnamnet till rätt ägare. Att bädda in `m.seat` som attribut vid renderingen löser detta utan att ändra serverns protokoll.

**Serverkompromiss: ingen unicast-logg utan serverändring.** Idealt vore att skicka en privat notering till avsändaren ("X targets Y (revealed to you)"). Det kräver ett nytt server-verb för unicast-loggning. Kompromissen: din privata notis + publik rad — avsändaren ser inte din privata rad, men den publika "X targets a revealed card." är tillräcklig information. Dokumenterat för framtida server-utökning.

**`clearAnyTarget` måste diskriminera på `cid`.** `setRTarget(null)` loggar "⌖ Target cleared." offentligt — rätt för remote targets med cid. Men för namnbaserade targets ska rensningen gå via `setNameTarget(null)` för konsekvent loggning. Därför: `if(!net.rtarget.cid) setNameTarget(null) else setRTarget(null)`.
## v1.88 learnings (logTo-verb)

**Unicast-logg utan att dela revealHand-verbets semantik.** `revealHand` tar ett array av kort och har komplex routing (all vs. specifik mottagare). En enkel privat loggrad behöver bara `{toSeat, html}` → ett separat minimalt verb. Håll server-verb fokuserade på ett ansvar.

**Safe degradation vid version-mismatch.** `logTo`-sends på klienten är `if(r.seat)` -guardade och `mpSend` skickar bara om `inRoom()`. Mot en gammal server (1.6) ignoreras verbet tyst — avsändaren missar sin privata notis men inget kraschar och den publika raden syns ändå. Dokumenterat i v1.88-journal.

## Phase 0 learnings (refactor groundwork: edge fixes, sweep #2, logic suite, ring buffer)

Most of the durable facts from this session are folded into **Technical learnings** / **Technical pitfalls** above (seats-as-indices + server-derived identity, the Node DOM-stub harness, the `dbg`/`elysiumDbg` channel, the wired-but-never-tested rot). The meta-lessons specific to the stretch:

**"Single-file" is two separable properties; only one is sacred.** The whole single-file-vs-build debate dissolved once we split it: the **artifact** being one double-clickable `file://` HTML is the real constraint (it's how Johan and his friends run it); having **no build step** is just developer convenience. Fork **B1** keeps the first and relaxes the second - the source can be flat script fragments sharing one scope, dumb-concatenated into the HTML, with a **byte-identical diff** of the built file against the current one as the safety gate (no import/export, no bundler, no behaviour change to prove). That reframing is what made the refactor tractable; the plan lives in `elysium-refactor-analysis.md`.

**Read the code, don't trust the plan.** The refactor analysis proposed a `baseView()` config-matrix test - but reading the function showed it had already been reduced to `return 'l2'` in v1.99.27. A matrix test would have asserted a contract that no longer exists. Every "obvious" test target got checked against the live source first; two (the `t` timestamp in `serializeGame`, and `buildPub` keeping face-down *names*) changed the assertions materially. When you can load the real code, load it before writing tests *about* it.

**Parked vs dead is a real distinction with different homes.** v1.99.27 *parked* revivable features -> archived verbatim to `kodarkivering.md` with breadcrumbs. This session's sweep removed dead developer scaffolding (`demoBoard`, `startDemoTable`, `applyL2Experiment`, `thump`, ...) that was never reachable and has no future -> deleted outright, *not* archived. Don't reflexively archive everything you delete; archive what someone might want back, delete cruft. Documenting *why it wasn't archived* is the part that keeps the next session from "restoring" it.

**A swallowed error is worse than a loud one when you can't watch the runtime.** The client's 46 empty `catch`es plus zero `console.*` meant an online desync left no trace at all. The ring buffer's value is precisely the *silent* paths - so it was wired into the network/escrow catches (where desyncs hide), not the benign ones (`c.el.remove()` is expected to miss). Capturing uncaught errors too (`window.error`/`unhandledrejection`) is free coverage. The buffer records unconditionally; the flag only controls live console mirroring - so you get history after the fact without having had debug on.

**Server `VERSION` is its own 1.x line, and the docs had drifted from it.** The code says `const VERSION='1.7'` (1.6 -> 1.7 -> 1.8 across sessions), but the v1.90-v1.99.26 prose called it "1.99.18". Because the client sends `v:VERSION` and the server compares to its own, the client/server numbers genuinely differ and the online mismatch warning fires - a state the journal had already accepted (online is secondary to hotseat). Lesson: when two components version independently, write the number from the code, not from the sibling's scheme, and say out loud whether the mismatch is intentional.

## v1.99.27 learnings (pre-2.0 dead-code sweep)

**"Park view L1" was impossible as literally asked - L1 is the base layer, not a sibling view.** L1/Simplified is `#board` with no `l2mode`/`l3mode` class; L2 and L3 are classes layered on top, and `exitL2`/`exitL3` just remove them, returning you to L1. `baseView()` also lands on L1 whenever Default-view = Simplified, `skipL2` is on, OR there's one live player (`opp ? 'l2' : null`). So removing the Solo entry point can't make L1 dead - it's reachable in hotseat and online. The honest move was to retire L1 as a *chosen* view (force Normal, drop the Simplified option) and flag full retirement (route one-player tables through the existing `l2solo`) as 2.0 work. **Lesson: before "parking a view", check whether it's a discrete mode or the base state the others are deviations from.**

**The shared-vs-feature boundary is the whole game when parking UI.** The Alt tray and Quick React strip *looked* self-contained, but their card renderers were load-bearing elsewhere: `avRenderCard` is used by the **Played** tab, and `qrCard`/`qrTap`/`qrHandDown`/`qrDragGive` (+ the `.qr*` CSS) are used by the **Ready** tab and the bottom **dock**. Only the `#altview`/`#qrbar` *containers* and their render/open/close functions were feature-specific. Reading each helper's call sites before cutting is what kept the tabs alive - a block-delete by name (`qrCard`, `.qrc`) would have broken Ready + dock. **Always grep every caller of a "feature" helper before parking it; shared leaf-renderers hide inside feature code.**

**Dangling references to a deleted function are latent, not always loud.** `avRenderCard` (kept) bound the now-deleted `avTap` inside an `if(interactive)` branch - but the only live caller (the Played tab) passes `interactive=null`, so the branch never runs and JS never resolves `avTap`. No crash, but a trap for future code. We cut the dead branch rather than leave the reference. **Lesson: when parking a function, also remove any binding to it inside kept code, even if that binding is currently unreachable.**

**`kodarkivering.md` convention.** Parked code is archived verbatim (extracted by the patch script itself, so it's byte-exact) with a what/why/where note per block; the live file leaves a one-line `// PARKED -> kodarkivering.md` breadcrumb at each site. Harmless leftovers (localStorage load-guards for retired settings, null-guarded listeners on removed elements, never-evaluated ternary branches) are intentionally left live and noted, rather than chased down.

**Self-inflicted assert trap.** A sanity assert `'qrOpen' not in s` tripped on the *PARKED comment itself* (which named the variable). When asserting an identifier is gone from a file you're also writing breadcrumbs into, keep the identifier token out of the breadcrumb text (or assert on a code-shaped pattern, not the bare word).

**Removing a setting is not the same as removing the view.** The first pass removed the Default-view setting and the Solo entry but left three navigation paths that could still land on L1: `baseView()` fell back to `null` for solo tables, `levelOrder()` included `null`, and `stepLevel(-1)` from a Visit went to `null`. All three had to be patched (`baseView`→always `'l2'`, `levelOrder`→`['l2','ov']`, Visit zoom-in→`'l2'`). The four `switchView(null)` calls that remain are game-teardown (leave/disconnect/stop-hotseat) — they're correct. **Lesson: when retiring a view, grep every path that enters it, not just the settings that select it.**

## v1.90–v1.99.26 learnings (L3 Overview build-out + the Edge token saga)

A long arc: building the L3 "helicopter view" pool zone, then giving the Edge a draggable physical token that hands off between players. Most of the hard-won facts are folded into **Technical learnings** and **Technical pitfalls** above (the L3 two-coordinate-worlds model, `l2pub` already carrying Z, geometric hit-testing, mat-relative fractions, the three hotseat layers, `pub.tokens` rendering, inverse-zoom, transition-glide). The meta-lessons specific to this stretch:

**The Edge token took ~10 iterations because the bug moved between layers — diagnose the whole pipeline, not the symptom.** The token "disappeared on handoff" had at least four distinct root causes hit one after another: (1) `elementsFromPoint` couldn't see the mat through the stage → handoff never fired; (2) `dropPos` read after `removeEdgeToken` → null; (3) `tok.x` stale during drag → wrong stored position; (4) the token simply wasn't *rendered* on opponents' mats at all (not in `pub.tokens`) — the deepest one, and the one Johan spotted by passing the turn and seeing nothing. **Lesson: when a fix "doesn't change anything", stop patching the same spot and instrument the actual data flow** (a couple of `console.log`s at each layer would have found #4 immediately). Don't out-think the runtime when you can't run it.

**Reverting a wrong architecture early is cheaper than nursing it.** v1.99.9 moved the token *into* `#l3stage` to get free pan/zoom; it cascaded (pointer-events inherited, drag math doubled, coords needed un-folding twice). v1.99.11 reverted the whole thing and instead left the token on `#board` and called `placeToken` from `applyL3Transform` — three lines, using the proven `pubXform` path. The clean solution was simpler than the workaround. When a change spawns a chain of follow-on fixes, that's the signal the approach is wrong, not that you need one more patch.

**Geometric/fraction-based placement is the reusable backbone for "put my thing on your board".** Card-give (`opponentSurfaceAt` + `rx/ry` fractions) was already the right pattern; the Edge token, the handoff, and the Ready-zone restriction (`readyZoneAt`) all converged onto it. Reach for it first for any cross-board interaction in any view.

**Authoritative actions must clear the unique resource everywhere.** The Edge is unique, so the *button* (`edgeTakeover`) trumps a prior drag-handoff by clearing edge+token across all stored boards, all pub views, online via an `edgeTake` broadcast — then re-seating one token. A "take it back" action that only sets *my* flag leaves a ghost on whoever I'd handed it to. Same shape will apply to any future unique token.

**Pixel-nudge requests need the element to behave correctly first.** Johan's `+42 → +32` Edge-button nudge "didn't show" — because the button had inverse-zoom, so its apparent position was already wrong/zoom-dependent. Once `scale(Z)` fixed the behaviour, the nudge landed visibly. Lesson: if a small position tweak seems to do nothing, suspect the element's transform/scaling is broken before assuming the value is too small.

**Server-bump vid nytt wire-verb.** `logTo` är ett nytt verb i nätverksprotokollet → server VERSION 1.6 → 1.7. Klienter mot fel serverversion ser mismatch-varningen. Client-only patchar (CSS/JS utan nya verbs) kräver ingen server-bump.


## Fas 1 learnings (handler-maps, modaler, B1-konkatenatorn)

- **Handler-map a dispatch, but don't flatten a *stepped* one.** `mpOnMsg` (client) and the server's `handle()` both became `{ verb: handler }` maps + a thin lookup, killing the long if/else drift (else-less = unknown verbs ignored silently, the prior behaviour). But the server's dispatch is **staged** — `create`/`join` run *before a room exists*, then the spectator phase, then `const room=conn.room, p=conn.player`. Only the **in-game** chain (45 verbs) became `GAME_HANDLERS`; the pre-room/spectator steps stay as pre-dispatch. Mapping the whole thing would have broken the lobby/spectator paths. (`chat` legitimately exists in two phases — spectator and in-game — and doesn't collide.)
- **The B1 concatenator: one scope, dumb concat, `cmp` is the law.** Source = flat `.js` fragments sharing **one** scope (no import/export), reassembled into the HTML through a `/*@@ELYSIUM_APP_JS@@*/` placeholder by a ~50-line zero-dep `elysium-build.js`. The **byte-identical `cmp`** against the previously delivered file is the only correctness gate. The single-file stays source-of-truth; `elysium-src/` is *derived* — so **re-split (from `/tmp/original.html`) + build + cmp after every client patch** to keep the fragments current and proven. Watch the trap: the splitter reports **character** counts, `ls` reports **bytes**; multibyte UTF-8 (emoji, em-dashes) makes them differ — that gap is a red herring, `cmp` is the truth. Leaf-module anchors must be unique plain-ASCII substrings (a header comment + the opening `function`/`const`), and each leaf is cut to start at its header and end at its closing `}`.
- **The modal system restores what was underneath.** `uiDialog`/`uiAlert`/`uiConfirm`/`uiPrompt` are Promise-based over the existing `#overlay`. The subtlety that makes them safe *inside* other overlays: `uiDialog` remembers the modal showing underneath (`shownModal()`) and **restores it on close**, so a confirm raised from inside the deck-lab doesn't dismiss the lab. Esc/Enter/backdrop are handled with capture + `stopPropagation` so they pre-empt the global backdrop-close. A top-level confirm (no enclosing function to make `async`, e.g. the init resume prompt) uses `.then()` instead of `await`.
- **Async ripple: don't make a hot, broadly return-consumed function `async`.** Turning a function `async` makes every `if(fn())` caller see a *Promise* (always truthy), so **every return-consuming caller must be awaited** — predicates especially (`_promptN`, `masterPlayGate`, `phaseLeaveOK` each had multiple callers, all verified awaited). Fire-and-forget callers survive only because an `async` function runs synchronously up to its first `await` (so the no-`await` common path stays sync). When only a rare dialog needs awaiting (the `move()` discard confirm), **lift the dialog to the UI/interaction layer** instead of asyncing the data function: `move` has 7 return-consuming `if(move(…))` callers + ~25 fire-and-forget in the drag hot path; asyncing it would ripple through all of them.
- **Lift UI confirms OUT of pure data functions.** A `confirm()` living inside a data mutation (`move`) couples the data layer to the UI and forces the whole function async. The fix pattern: move the confirm to the **interaction layer** where the user action originates (the drop handler, the menu item) via a small `async` gate (`discardGateOK`), keeping the data function synchronous + pure. Bonus dividend: the gate naturally scopes the warning to the *discretionary* single-card entries, so forced/batch paths (random discards, multi-card moves) correctly skip a prompt that would be annoying or nonsensical there.
- **`keepLog` must gate `turnLogs` in BOTH `clearTable` AND `restoreGame`.** The hotseat "one shared table log" only holds if the turn-*history* (`state.turnLogs`) survives a seat swap. Both functions reset/overwrite `turnLogs` ungated by default, while `curLines` was already gated — so each `setActivePlayer`→`restoreGame({keepLog:true})` clobbered the shared history with the incoming seat's (empty) blob, and only the creator's turns survived a lap. Gate both. (Live `#log` shows the current turn and is cleared each pass *by design* — short and scrollable; the turn-dropdown is the shared full history.)

## Fas 2 learnings (löv/feature-extraktion, helpers-registry, core-sömmar)

- **Registry-mönstret (HELPER_DEFS) — driv data + wiring från en array, lämna gating ifred.** När en nyckel-mängd räknas upp på flera ställen (defaults, persistens, UI-id:n, handlare) → en `[{key, el, …}]`-array som GENERERAR alla. Men gränsdra: registret driver *data och bindning*; *logiken* (här `hx()` + policy-gating) lämnas orörd och läser bara `obj[key]`. Rika UI-etiketter bor kvar i markup, inte i JS-strängar (läsbarhet). Lägg-till = en rad. Samma mönster som TOKEN_DEFS.
- **Klient↔server-uppräkningsdrift är exakt det registret ska fånga — och den fanns redan.** Serverns `HELPER_KEYS` saknade `hand`+`handReminder` (6 av 8) → låsta online-bord propagerade inte dem. Klassisk "glömt ett ställe när jag la till en feature". Delade filer (klient/server) kan inte dela en bokstavlig array, så: spegla listorna + kommentera kopplingen i båda (`mirror the client's HELPER_DEFS`). Lärdom: varje gång en uppräkning dyker upp i mer än en fil, misstänk drift.
- **Koden är skriven i ANVÄNDNINGS-ordning, inte modul-ordning.** Detta avgör vad som går att extrahera byte-identiskt: **löv och features** (sfx, fx, tokens, decklab, hotseat) råkar vara sammanhängande → klippbara; **core-sömmar** (state, render, input) är utspridda genom hela filen → INTE klippbara utan att flytta funktioner. Kartlägg alltid funktions-positionerna (`grep -n "function X"`) FÖRE du lovar en extraktion. `net` är den enda rena core-sömmen (men bisekterad av esc-crypto-lövets placering — net-headern fastnar i föregående fragment). `hotseat` är ett rent block (offline-kärna + setup-fönster sammanhängande; bara `giveHot` ligger separat).
- **Relokering (att flytta funktioner för att gruppera) är säkrare än det låter — risken sitter i top-level, inte funktionerna.** `function`-deklarationer hissas → flyttbara var som helst, beteende-säkert. Risken är 100% i top-level-satser (const/let TDZ + sidoeffekter i load-ordning, ~250 st). Disciplin-regel: flytta BARA funktions-deklarationer; rör ALDRIG top-level-satsernas inbördes ordning. (Full plan: `elysium-relocation-brief.md`.)
- **Byte-identitet och relokering är ömsesidigt uteslutande.** `cmp` är grinden för extraktioner (ren diff = programmet bokstavligen oförändrat). I samma sekund du flyttar kod för modulisering är du medvetet inte längre identisk → verifiera som vanlig redigering (node --check + svit + live-test). Därför är relokeringen SIST, med testnätet tjockt. Bygg `state`-kontraktets round-trip-tester (serialize/restore/buildPub) FÖRE state relokeras.
- **Byggarens "(N bytes)"-utskrift är TECKEN, inte byte.** `elysium-build.js` loggar `out.length` på en UTF-8-sträng; multibyte-glyfer (em-dash, ikoner) blåser upp den verkliga byte-storleken. En "625395 bytes"-utskrift mot en 627479-byte-fil är inte en diskrepans — det är tecken vs byte. `cmp`/`wc -c` är byte-auktoriteterna; lita aldrig på den utskrivna siffran. (Samma multibyte-villospår som splitter:ns teckenrapport.)
- **Splitter-disciplin för flera fragment:** ankare på plain-ASCII (sektionskommentar + öppnande `function`/`const`); assertera unikhet (`js.count(anchor)==1`) FÖRE indexering; assertera källordning (`p0 < p1 <= p2 …`); assertera reassembly (`a+b+c…==js`) internt; städa stale fragment (`glob`+`os.remove`) före skrivning så renumrering inte lämnar föräldralösa filer. Manifest-driven build → ny ordning = bara manifest-rad.
- **Färsk dead-code-scan: funktioner som bara förekommer 1 gång = död.** Bortom den betrodda §2-inventeringen (som var slut-rensad): extrahera alla `function NAME(`-namn, räkna `\bNAME\b` i hela filen; `count==1` betyder att namnet bara står vid definitionen → inte anropat, inte ens via sträng-dispatch (det skulle ge en andra förekomst). Gotchas: (1) **en-radare med trailing-kommentar** (`function sfxLock(){ … }   // swosh`) — detektera en-radare via brace-balans på def-raden (`count('{')==count('}')>0`), inte via "slutar på `}`" (kommentaren bryter det). (2) **`\b` mot prefix-kollisioner** — `function burnRandomAsh\b` matchar INTE `burnRandomAshN(` (h↔N är ingen ordgräns), `sfxPlay\b` inte `sfxPlayCard`. Avgörande för att inte ta bort fel funktion. (3) **Vetta före borttagning** — count==1 bevisar att den är död, men *varför* den finns är en produkt-fråga: superseded helper (`cardColor`), tidig version ersatt av bättre (`sfxLock` → dedikerat sfx; `burnRandomAsh` → `burnRandomAshN`), eller slopad feature (`giveCardTo` → give-control). Flagga för Johan; tvinga inte borttagning av det som kan vara avsedd-men-ej-kopplad scaffolding.
- **Dead-code-borttagning är beteende-neutral → ingen versionsbump, men en riktig filändring.** Koden anropades aldrig, så runtime är oförändrat (samma 1.99.37), men filen krymper. Verifiera ändå split→build→cmp (byte-identisk reassembly bevisar att splitter-ankarna inte rubbats — döda funktioner låg inuti fragment, inte vid gränser) + svit + node-check. Borttagning i **omvänd radordning** (högst rad först) så index inte glider; assertera `count==0` efter + att keep-symboler överlevde.
- **State-kontrakt-testerna är relokeringens skyddsnät — pinna det en omflyttning kan bryta i tysthet.** Bortom idempotens (`serialize→restore→serialize`): **single-zone-invarianten** (`move` håller kortet i exakt en zon), **attachment round-trip** (`host.attached`/`child.host` överlever — attachment-state är komplext nog att förtjäna ett eget test), **id/position-överlevnad**, **buildPub-renhet** (stabil + muterar inte). Harness:en har en *curated* expose-lista (`new Function`-fabrik som returnerar valda globaler i scope) — nya tester som rör interna funktioner (`attach`/`place`) kräver att de läggs till i listan. Faithful path > manuell wiring: `attach()` kördes rent under DOM-stubben, så testet motionerar den *riktiga* koden, inte en handbyggd state.

## Relokeringslearnings (concern-gruppering av monoliten)

- **Brace-counting-parsern måste hantera default-param-braces.** `function move(c, zone, opts={}){` har tre `{`-tecken: `opts`-öppning, `opts`-stängning, och funktionskroppens öppning. En naiv "depth == 0 = slut"-logik stoppar vid `opts}`-stängningen och rapporterar funktionen som en enradare. Lösningen: kontrollera depth vid **radens slut** (hela raden processad), inte mitt i. På en rad som `opts={}){\n` slutar depth på 1 (kroppen öppen) — korrekt, fortsätt. På en genuin enradare (`function goHome(){ ... }`) slutar depth på 0 — korrekt, rapportera som komplett. String/comment-awareness (track `in_str`, `in_lc`, `in_bc`) behövs också — braces inuti strängar och kommentarer ska inte räknas.
- **`async function`-deklarationer hissas identiskt med `function`-deklarationer.** Det finns 28 `async function` top-level i Elysium (inkl. `dropCard`). De hissas, de kan flyttas fritt, de följer disciplinregeln. Briefen nämnde bara `function` — det är säkert men kan vilseleda till att `async function` verkar icke-hissade.
- **Input-concernen är ihålig under disciplinregeln.** Input-logiken lever i 24+ frusna inline `addEventListener`-callbacks (board/document/window) — inte i de ~10 namngivna funktionerna (`bindCard`, `tap`, `dropCard` etc.) som kan flyttas. En `input.js` under disciplinregeln samlar funktionerna men inte beteendet. Konsekvens: (1) en fullständig input-modul kräver handler-body-extraktion (ett beteende-refaktor, utanför disciplinregeln); (2) touch-roadmapen (rörelsetrösklar, time-gates) kan inte uppfyllas av relokeringen ensam; (3) en tunn gruppering i glue med sektionskommentar (alternativ C) är mer ärlig än ett eget fragment.
- **`drag`/`giveDrag`/`tdrag` är `let`-satser, inte funktioner.** Briefen listade dem under input som om de var flyttbara. De är top-level `let`-deklarationer (rad 2621/5605/8776) som hissas INTE → fryses per disciplinregeln. Detsamma gäller `let pushTimer` (7316), `let escrowTimer` (7391), etc. **Grepa alltid mönstret innan du lovar en flytt.**
- **Splitter-ankare som pekar på funktioner som relokeras bryts — planera bytet FÖRE flytten.** `NET_END = "function buildPub(){"` försvinner från sin position när `buildPub` relokeras. Ersätt med en **fryst** deklaration i regionen som aldrig kan flytta: `let escrowTimer=null, lastHandSig='';` (unikt, top-level `let`, permanent). Byt ankaret i `split_client.py` INNAN du kör relokeringsskriptet, inte efter.
- **Tre oberoende utvärderingar > en.** Att granska samma brief med tre perspektiv avslöjade fem konkreta fynd som ingen ensam utvärdering fångade: specifika `let`-deklarationer i flytt-zonen (utv. 2), NET_END-ankarbrott + PUBZ-ersättare (utv. 2), input-modulens ihålighet + touch-roadmap-frikoppling (utv. 3), att drag/tdrag/giveDrag är `let`-satser (utv. 3), och `qrDragGive` som en input-funktion i render-regionen (utv. 1).
- **Escrow → net.js är TDZ-säkert trots att `let escrowTimer` stannar efter.** `pushEscrow` (som flyttas in i net-kroppen) refererar `escrowTimer` (en `let` som stannar i glue, textmässigt EFTER net). Eftersom `pushEscrow` är en hoisted `function`-deklaration refererar den `escrowTimer` bara i sin kropp (vid runtime), aldrig vid definitionstid. Alla anrop till `pushEscrow` sker vid runtime (veriferat: bara från `schedulePush`, också en hoisted fn) → `escrowTimer` är initialiserad långt innan. **Regeln:** en hoisted funktion kan textmässigt föregå en `let` den läser, förutsatt att den ALDRIG anropas vid load-tid.

## Board Mode learnings (L4 Classic / Structured)

- **Board Mode är ett spelläge, inte en vy.** `net.boardMode` sätts vid spelstart och styr vilka vyer som finns — det är INTE en view i `net.view`. Classic (`'classic'`) = en enda vy (L4), Structured (`'structured'`) = L2+L3 som tidigare. De är ömsesidigt exklusiva och kan inte kombineras.
- **"Classic" = Lackey-spelarens perspektiv.** Det fria bordet är det klassiska sättet att spela digitalt; Elysiums zoner och helpers är innovationen. Namngivningen speglar detta medvetet.
- **`freeBoard()` är gate-funktionen — inte `l4on()`.** `freeBoard()` testar `net.boardMode` (persistent under hela spelet). `l4on()` testar `board.classList.contains('l4mode')` (transient, sätts av enterL4). Använd `freeBoard()` för att gata beteende som gäller hela spelet (auto-deal, menyval), `l4on()` för rendering/layout.
- **L4 + Tournament Mode = ren Lackey-upplevelse.** Tournament Mode släcker alla helpers; L4 tar bort zonerna. Tillsammans ger de en frihandsupplevelse med bättre fidelity.
- **Server validerar boardMode strikt.** Bara `'classic'` accepteras; allt annat → `'structured'`. Samma mönster som `tableView`/`l3shape`.
- **`cycleView()` och `stepLevel()` är no-ops i Classic** — `levelOrder()` returnerar en single-element array `['l4']`, så stepping clampar till samma index och cycling returnar direkt.

- **L4 bygger PÅ L3, inte bredvid.** `enterL4()` sätter `l2mode`+`l3mode`+`l4mode`. Det gör att `l3on()===true`, så ALL L3-infrastruktur (zoom/pan, dock, hand-hover, scroll-zoom, drag-pan, `applyL3Transform`, `setCenterFrameA`) fungerar utan modifiering. L4 = L3 minus mattor plus hela filtets yta som slot. Skillnaden gatas med `l4on()` (CSS `#board.l4mode`). Att bygga L4 som en helt separat vy hade krävt duplicering av ~500 rader zoom/pan/dock-logik.
- **`l3myslot` = hela filtet i L4.** `setCenterFrameA(g.ax, g.ay, g.aw, g.ah)` sätter up `l2pub` för full-board-mapping. Kort i `ready` placeras via `pubXform(c.x, c.y)` med `l2pub.s = baseS*Z` — exakt samma pipeline som L3-slottar. `applyL3Transform()` anropar `setCenterFrameA(l3myslot)` vid varje zoom/pan-ändring.

- **Pool-globerna är INTE tokens.** De använder ett eget system (`l4Globes` Map) parallellt med `tokens` Map. Anledningen: pool-glober har per-seat-instanser, visar extern data (`state.pool` / `pub.pool`), har scroll-wheel-interaktion, och skiljer sig i ägande-modell (din är interaktiv, motståndares är read-only). Att stoppa dem i `TOKEN_DEFS` hade krävt modifiering av den generella `makeToken`/`bindTokenDrag`-pipelinen. Placering (`placeL4Globe`) använder dock samma `pubXform`/`pubInv`-pipeline som tokens.
- **I L4 delas bara library ut — crypt stannar i högen.** Gaten `if(!freeBoard())` runt crypt-loopen i `dealOpeningSync`/`dealOpening` + serverns `dealNow(p, room)`. Library-kort (7 st till hand) delas ut normalt. Spelaren drar crypt-kort manuellt via docken. Servern fick `room` som ny parameter till `dealNow` — alla tre callers uppdaterade.
- **`layoutZone('torpor'/'uncontrolled')` i L4 = fri placering.** En `if(l4on())` gren som behandlar dem precis som ready-zonen: `place(c, c.x, c.y, ...)`. Nydragda kort med `x=0, y=0` får en slumpmässig startposition nära center. Det gör att kort i alla tre spel-zoner beter sig identiskt i L4 — inga auto-arrangerade rader.
- **`pcoHome` skippar i L4 — `conv.pco` räcker.** `showOwnerOutline` har villkoret `!freeBoard()` runt `pcoHome`-checken. I L4 finns ingen hemzon (alla kort är på samma yta), så `pcoHome` (som gömmer DINA outlines i DIN hemzon) är meningslöst. `conv.pco` hanterar allt: `'on'` = alla, `'mine'` = göm mina, `'all'` = göm alla.
- **`poolGradient(hex)` skapar en konsekvent sfärisk gradient.** Tre steg: highlight (30% mot vitt), bas, shadow (65% mot svart). Används av `#pool`, `#dockPool`, L4-glober och mat-glober. CSS-gradienten är fallback; inline `background` style överridear den.
- **L4 piggybackar `l3mode` → alla `l3on()` guards som triggar `renderL3()` måste ha `!l4on()`.** Tre ställen hittades: `edgeTakeover`, `hostSetPoolHot`, `hostOustHot`. Utan guarden skapar `renderL3()` opponent-mattor på L4-bordet. Calls som kollar `net.view==='ov'` istället för `l3on()` är säkra (L4 har `net.view==='l4'`, inte `'ov'`).
- **L4 element scaling kräver konsekvent `L4_CARD_S`/`L4_TOKEN_S` överallt i drag-pipelinen.** Förutom `place()` måste multiplikatorn appliceras i: `regrabAtScale` (grab-offset vid scale-transition), drag-move `sc` (render under drag), gruppdrag `ms`, `dropCard`/`groupDrop` effektiv kortstorlek (`ecw/ech`), och ghost-element scaling. Utelämning → kort hoppar vid drag-start, snappar mot kanten, eller renderas i fel storlek.
- **Zoom-clamp fångade initialt vår L4-zoom.** `l3zoomTo` klampade till `Math.max(0.6, ...)` och slidern hade `min="0.6"`. Vår `l3zoom=0.45` ignorerades tyst → element renderades vid zoom 0.6 istället. Fix: sänkte min till 0.2. Lärdom: kontrollera alltid att ett satt värde inte clampas bort av en annan kodväg.
- **Ghost-element (position:fixed) måste skalas med `fitScale × l2pub.s × L4_CARD_S`** för att matcha kortstorlek på bordet. `fitScale = board.getBoundingClientRect().width / board.clientWidth`. Drop-offset: `CW × l2pub.s × L4_CARD_S / 2` istf `CW/2`.
- **`dropZone()` i L4 måste bypassas** — de osynliga torpor/uncontrolled-zonboxarna (gömda av CSS men fortfarande positionerade med getBoundingClientRect) fångar drops i nedre delen. Fix: i L4, bara dock-zoner hit-testar; allt annat → `ready`.
- **Dock tvångsstängning var onödig** — `setL2Dock(false)` vid drag-start blockerade den naturliga hover-logiken (som redan hanterar öppna/stänga baserat på musposition). Borttagning → docken beter sig korrekt: stängs när kortet lämnar, öppnas när det återvänder.
- **`c.converted` flag behövs för revert-meny.** `c.kind` ändras vid konvertering, men utan `c.converted` kan vi inte veta om kortet ursprungligen var library. Flaggan persisteras i snapshot (undo) + serializeGame (save) + buildPub (online sync).
- **KRCG Path-fält (PR #830, OPEN):** `Card.path` exponerar V5 Sabbat path från VEKN CSV "Path"-kolumnen. Fältet serialiseras till `vtes.json` som `"path":"Path of Caine"` etc. (`None`/utelämnat för icke-Sabbat). PR av Spigushe, öppnad 25 juni 2026. Elysium läser `card.path` i `loadCardInfo()` och använder det för att blockera manuellt path-val.
- **Path vs Clan prioritet för pool-globikon:** `displaySymbol()` returnerar path-ikon om antalet path-följare strikt överstiger antalet av den största klanen. Vid lika → klan vinner. Manuell override (`chosenClan`/`chosenPath`) trumfar alltid. `clanSymUrl()` hanterar path-namn via `PATH_SVG_SLUGS` (normClan→slug-mapping).
- **Sabbat-detektion i VTES: deriverad vid laddning, inte vid varje menyöppning.** `loadCardDB()` deriverar `ri.sect` från `card_text` under parsningen (Sabbat/Camarilla/Anarch/Laibon/Independent via `\bX\b` regex). `card.sect` (om KRCG serialiserar det i framtiden) trumfar som override. Resultatet: menykoden behöver bara kolla `ri.sect==='Sabbat'` — inga regex, inget fallback-mönster. Derivering i parser-loopen istället för i varje menyöppning är billigare och enklare att resonera om.
- **En meny för glob-ikon, inte två.** Klaner och paths listas i samma "Choose clan" submeny — paths visas under en separator om det finns path-vampyrer. `state.chosenClan` håller det manuella valet (kan vara klan ELLER path-namn). `state.chosenPath` existerar inte — `clanSymUrl()` hanterar path-namn via `PATH_SVG_SLUGS`, så renderers behöver inte veta skillnaden.
- **`updateMyClan()` hanterar nu BÅDE klan och path-detektion.** Det är det enda stället som anropar `detectClan()` + `detectPath()` + `refreshClanDisplay()`. Alla anropare (phase change, card move, chosenClan-reset, convertCardKind) behöver inte ändras.
- **`displaySymbol()` är vad pool-globen visar; `activeClan()` är fortfarande den rena klan-detektionen.** Dessa är separata: `displaySymbol` kan returnera en path-sträng istf en klan. `activeClan()` oförändrad — används i klanspecifik logik (Choose Clan-meny, buildPub, klan-logg). Alla renderers (matglobar, L3-glober, stat-rader, pname-prefix) passerar värdet genom `clanSymUrl()` som hanterar båda.
- **pathBlood helper gates till phase 3 (Influence).** `disabled: !_phaseOK` på menyalternativen sätter CSS `.mi.disabled` (opacity .4, `pointer-events:none`) och hoppar över click-handler i `renderItems()`. Disabled-stöd lades till i `renderItems()` som del av path-implementationen — det saknades helt innan.

## v2.2 learnings (Path of Enlightenment, L4 polish)

- **Pile drag target functions are separate from `dropZone()`.** `libDragTarget()` and `cryptDragTarget()` have their OWN zone hit-testing (`#z-ready.getBoundingClientRect()`), independent of the card-drag `dropZone()` function. In L4, `dropZone()` correctly returns `'ready'` for the whole board, but the pile-drag functions didn't — they hit-tested the positioned `#z-ready` box which is smaller than the board in L4. Fix: add `board.getBoundingClientRect()` as L4 fallback. Lesson: when two drag pipelines target the same zones, they must agree on hit-testing logic.
- **`detectClan()` must skip face-down cards — same as all other detection functions.** `detectPath()`, `clanCount()`, `pathCount()` all filtered `c.faceDown`, but `detectClan()` didn't. Result: drawing crypt cards face-down to uncontrolled leaked clan info to the pool globe. A consistency check across all detection functions would have caught this.
- **Pool globe hint: show-once pattern.** The "Drag your pool globe" hint was recreated on every `renderL4Globes()` call (including panel-triggered resize). Fix: `l4hintDone` flag, set `true` on first drag, checked before creating the hint. Reset only in `enterL4()`. General pattern: any one-time UI hint in a re-renderable function needs a dismiss flag.
- **"Special…" submenu consolidates non-generic actions.** Crypt-specific (Follow Path) and library-specific (Convert to vampire) actions live under one "Special…" submenu at the bottom of the card menu. The submenu only appears if there are items (`_specSub.length` check). This keeps the main menu clean and extensible — future card-type-specific actions go here too.
- **`chosenClan` can hold a path name.** Since `clanSymUrl()` handles both clan and path names via `PATH_SVG_SLUGS`, there's no need for a separate `chosenPath` state variable. One override field, one menu, one codepath. Simpler.
- **Click-outside-to-close needs an exclusion list.** The zoom panel close listener must exclude clicks on the panel itself AND the toggle button (otherwise the toggle's own click opens the panel and the document listener immediately closes it). Pattern: `if(ctl.contains(e.target) || tog.contains(e.target)) return;`.

## Reflektiv analys — kodmognad efter refaktorn

### Best practice
Starkt. Handler-map dispatch, Promise-modaler, HELPER_DEFS-registret, B1 med 13 fragment, concern-grupperad monolith. Den största kvarvarande skulden: 24+ inline `addEventListener`-callbacks borde extraheras till namngivna funktioner (testbarhet + touch-roadmap-förutsättning). Det är ett naturligt Fas 3-projekt som bygger PÅ den grupperade koden.

### Funktionalitet
Komplett för kärnspelet. Solo, hotseat, online, VDB-import, KRCG-kortdata, ägar/kontrollant, oust/bounty, klan-detektion, counters/tokens, hand-escrow-krypto, komplett logg. Roadmap-items (touch, vote-helper, fritt L3-zoom, face-down cross-plays, leading-edge push, hotseat-djup, server-auktoritativ rund/turräknare) blockeras inte av refaktorn.

### Prestanda
Mer än tillräckligt för VTES. Noll-beroendefilosofi = noll framework-overhead. VTES-skalor (5–25 kort i play) gör JS-logikens kostnad försumbar. De potentiellt dyraste hot paths (`opponentSurfaceAt`/`readyZoneAt` med live `querySelectorAll` + `getBoundingClientRect` i pointermove) är medvetet geometriska (robust över stacking contexts). `buildPub` och `schedulePush` (600ms trailing debounce) skalas linjärt med korträkning. Prematur optimering (DOM-diffing, buildPub-cache, rAF-batching) vore mer komplexitet än vinst.

### Driftsäkerhet
State-kontraktet (16 assertions) vaktar de kritiska invarianterna. Server-dispatch (3 assertions) + ~195 i de fullständiga server-sviterna. Debug ring buffer (`dbg`/`elysiumDbg`) fångar tysta fel. Defensiv `restoreGame`. Global error-fångst. **Det stora gapet:** render/interaktion har noll tester — Johans live-test är den enda grinden. Två mindre skulder: serverns `writeFileSync` blockerar event-loopen (osynligt vid 1–2 rum, riskabelt vid skalning), och ingen rate-limiting klient→server (acceptabelt i "vänner runt ett bord"-modellen).

### Framåtblick — mest värdefulla nästa steg
1. **Handler-body-extraktion** (Fas 3) — extrahera inline callback-kroppar till namngivna funktioner. Förutsättning för testbar input-logik + touch-roadmapen.
3. **Render/interaktionstester** — det största driftsäkerhetsgapet. Möjligt nu tack vare concern-grupperingen.
4. **Touch-stöd** — separat arbetsström, bygger på handler-body-extraktionen.
5. **Vote/referendum-helper** — nästa feature-modul.

## v2.3 learnings (Files, caching, UX polish)

- **`conv.imgCache` as a boolean guard in `imgUrl()` is the right pattern.** A Map.get on an empty Map is nanoseconds, but the boolean guard makes the intent explicit: "I have not opted into local caching, so don't even try." Zero overhead when off, minimal overhead when on.
- **IndexedDB keys for non-image metadata should use `_` prefix.** `warmImgCache` and `imgDbStats` skip `_`-prefixed keys, so the `_vtesJson` card database cache lives in the same `cards` store without polluting image counts or creating invalid blob URLs.
- **SVG icons use `svg:clan/slug` / `svg:path/slug` keys in the same IndexedDB store.** No schema change needed. The `svg:` prefix distinguishes them from card images (which are pure alphanumeric from `norm()`). `downloadImages` handles both via a simple `n.startsWith('svg:')` URL dispatch.
- **`.set-row[hidden]{display:none}` is required.** CSS `display:flex` on `.set-row` overrides the HTML `hidden` attribute's `display:none`. This is a common CSS-vs-HTML gotcha — any explicit `display` in CSS wins over `[hidden]`. The fix is a specificity-matched rule.
- **`loadCardDB` should separate data processing from data fetching.** `processCardData(data, src)` is the shared pipeline; `fetchVtesJson()` handles KRCG network; `loadCardDB()` orchestrates cache-vs-network. This enables both offline-first (IndexedDB cache) and online-sync (KRCG fetch) with the same processing code.
- **Image sync is lightweight: Set iteration + Map.has.** `imgSyncCheck` iterates `cardImageNames` (~4500 primary names) and checks `imgBlobCache.has()` — pure in-memory, no I/O, milliseconds. Only the delta (new cards) triggers downloads. Safe to run on every card DB load.
- **The `warmImgCache` cursor approach is memory-efficient.** IndexedDB cursors process one entry at a time. `URL.createObjectURL(blob)` creates a string reference (~50 bytes) — the blob data stays in the browser's blob storage (often disk-backed), not in the JS heap. 5000 blob URLs ≈ 500 KB of JS heap.
- **Download progress: hidden attribute needs CSS support.** The `#imgProgressRow` used `hidden` but `.set-row{display:flex}` overrode it. Pattern: any element with a CSS `display` rule needs a corresponding `[hidden]` rule.
- **`--tab-mid` offset is constant regardless of `#phases` width.** With two `flex:1` spacers, the center of `#phases` depends only on the fixed-width left and right groups, not on `#phases`' own width. The offset is `(rightGroup - leftGroup) / 2` — independent of how many phase buttons are visible.
- **Merging a boolean setting into a dropdown works cleanly with migration.** `conv.pcoHome` → `conv.pco='home'`. A single migration line in the conv loader (`if s.pcoHome && pco==='on' → pco='home'`) handles old saved settings. The old property name never appears in new code except the migration.
- **The `_lastBoardHover` tracker belongs in the existing `pointerover`/`pointerout` handlers.** No new listener needed — just an extra variable set alongside the stack-expand logic. Card elements always live in `#board` regardless of view (L1–L4), so the same handler works everywhere.
- **`cacheVtesJson` should store `size: raw.length * 2`** (JS strings are UTF-16, 2 bytes per char) so `imgDbStats` reports accurate storage usage.
- **`imgClearAll` → `localStorage.removeItem` order matters in "nuke all".** `imgClearAll()` calls `saveConv()` internally (sets `imgCache=false`), which recreates the `elysium.convenience` key. Running localStorage wipe AFTER ensures the leaked key is cleaned up.
- **HTML elements rendered before `<script>` must start hidden if the script decides their visibility.** `#empty` had `display:grid` in CSS — the browser rendered the welcome dialog while parsing HTML, before the script's `restoreGame()` at the end of `<script>` could hide it. Fix: CSS default `display:none`, explicit show after the startup restore attempt. General rule: any element whose visibility depends on JS state should start hidden and be revealed by the script.
- **DOM cleanup functions must remove ALL child elements they create.** `clearL4Globes()` removed globe elements and `#l4oppWrap` but forgot `#l4hint` — the hint stayed in the DOM when switching from L4 (Classic) to L2/L3 (Structured), visible as an orphaned floating label. Pattern: if a function creates dynamic DOM elements, its teardown must remove every one of them.
- **L0 must be exited BEFORE any `layout()` that involves cards.** `layoutZone()` calls `zoneRect()` which reads `getBoundingClientRect()` — zones hidden by `display:none !important` (L0) return zero dimensions. So `board.classList.remove('l0mode')` must happen before `clearTable()` → `resetForDeal()` → `layout()` in any game-start path. Enter L0 only when `state.cards.size === 0` is guaranteed (startup, after `clearTable` in teardown).
- **Batch downloads should collect failed names and retry once after a pause.** Transient network errors (rate limits, DNS, timeouts) often resolve in seconds. A 2-second wait + single retry pass within the same `downloadImages` call recovers most failures without user intervention. Permanent 404s fail on both passes and appear in the final count. The `imgBlobCache.has(n)` guard in the main loop already makes "Download all" idempotent — clicking it again skips cached images and only attempts missing ones — so it remains a manual fallback.
- **Download name sets must match the image server's filename scheme.** `cardInfo` stores three name variants per card (`card.name`, `card._name`, `card.printed_name`) for lookup flexibility, but KRCG hosts images only under `norm(card.name)`. Using `cardInfo.keys()` for downloads inflated the count ~30% with alias names that 404. Fix: separate `cardImageNames` Set (primary names only) for download/sync; `cardInfo` keeps all variants for lookup.
- **All game-teardown paths must enter L0.** `mpLeave()` (with game), `resetTable()`, and `leaveHotseat()` all result in an empty table — each must call `enterL0()` to show the clean felt surface. Missing one leaves the board in a raw L1-like state with empty zones visible.
